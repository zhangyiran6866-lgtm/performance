# Git 提交与推送规范 (Git Commit & Push Convention)

这是一份关于如何规范地进行 Git 提交以及向远端推送的操作准则。在本项目开发中，AI 助理必须严格遵守以下规则。

## 1. 提交信息规范 (Commit Message Convention)

每次进行 `git commit` 操作时，如果用户给出了提交内容，AI 必须根据本次代码的变动类型，**自动在用户给定的内容前面补充标准的提交规范前缀**（类似于 Angular 规范）。

**常用的前缀类型包括：**

- `feat:` 新增功能 (Feature)
- `fix:` 修复缺陷，修复 Bug (Bugfix)
- `docs:` 仅仅修改了文档页面/注释等相关的变更 (Documentation)
- `style:` 代码格式修改（不影响语义的变更，如空格，缩进，分号等）
- `refactor:` 代码重构（既不修复 Bug，也不引入新功能的代码变更）
- `perf:` 优化相关，如提升性能、体验 (Performance)
- `test:` 增加测试代码或更正现有的测试 (Testing)
- `build:` 影响构建系统或外部依赖项的更改（如 webpack, npm 配置）
- `ci:` 更改了 CI 配置文件或脚本
- `chore:` 其他非核心代码、日常事务相关的构建过程或辅助工具变动
- `revert:` 撤销先前的提交

**格式示例：**

- 用户提供了 `登录页面及功能实现` 并且是新功能，那么实际提交的 message 应为：`feat: 登录页面及功能实现`
- 修复了样式报错：`fix: 解决顶部导航栏样式遮挡问题`

## 2. 远端推送规范 (Remote Push Convention)

在执行代码推送操作 (`git push`) 时，**绝对不允许出现跨分支推送的情况**（即不可将分支 A 推送到分支 B）。

### 核心规则：

1. 请永远在当前分支，推送到远程的同名关联分支。
2. 建议使用的安全推送命令如下，使用 `HEAD` 显式推送到远程的当前对应的同名分支，而非盲目地进行 `git push`（如果没有关联远端可能会失败）或者是用有跨分支隐患的指令：
   在使用推送到远端时，AI 应该获取当前分支名称，并使用指令 `git push origin <当前分支名称>` 或者安全的 `git push origin HEAD` 保证同名推送。如果远端尚未建立该分支的跟踪，则应使用 `git push -u origin <当前分支名称>`。

---

> **给 AI 的严格指令：**
>
> - 任何时候只要进行提交操作，**必须**自动套用本提交规范判断出正确的标签（如 `feat`, `fix`, `chore` 等），不可遗漏。
> - 向远端推送前，**必须**确保推送目标指向远程同名关联分支，坚决防止出现覆写其他环境分支的情况。
